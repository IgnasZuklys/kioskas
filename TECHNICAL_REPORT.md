# Technical report — KPSGroup ticket platform

## System structure & chosen technologies (≤ 0.5 page)

The system implements the **3-tier multi-layer architectural style** as three separate .NET projects plus a React SPA:

- **Presentation tier** — React 18 SPA (`frontend/`) communicates over JSON/REST with `TicketPlatform.Api` (ASP.NET Core 10 controllers, JWT bearer auth, CORS for the dev origin, action filters). React handles all view state; the API has no server-side session.
- **Business Logic tier** — `TicketPlatform.Business`: use-case services (`AuthService`, `EventService`, `OrderService`), Strategy implementations for pricing, a Decorator chain for payment processing, and a `BackgroundService` consuming an in-memory channel for asynchronous email "sending".
- **Data Access tier** — `TicketPlatform.Data`: EF Core 10 (Npgsql) `AppDbContext` for ORM access; a Dapper-based `EventSearchRepository` for read-only event listings (Data Mapper). PostgreSQL 16 is the storage engine; concurrency uses the PostgreSQL system column `xmin` as an EF Core concurrency token.

The Business tier references only Data; the Api tier references both. Frontend talks only to the Api through a Vite dev proxy. No tier reaches across layers. JWT keeps the API stateless so multiple browser tabs of the same user account work independently.
